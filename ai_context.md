## 1. Сетка частотных доменов (Clock Domains)

Все тактовые частоты генерируются специализированным блоком PLL. Параметры частот и их фазовые соотношения зафиксированы во внешнем `.sdc` (Synopsys Design Constraints) файле автомата временного анализа.

| Имя тактовой частоты | Частота | Описание домена |
| :--- | :--- | :--- |
| sys_clk | 80 MHz | Системный домен. Команды управления, интерфейсы чтения данных |
| adc_clk | 65 MHz | Домен АЦП-1 потключеного к VGA усилителю. Скоростной захват сигналов, накопление данных |
| log_clk | 25 MHz | Домен АЦП-2 подключеного к логарифмическому усилителю |
| dac_clk | 50 MHz | Домен ЦАП. Генерация напряжений ВАРУ (управление трактом AD604) |
| hi_clk | 250 MHz | Высокочастотный домен для субтактового управления импульсом |

## 2. Дефайны, имена портов и правила синхронизации

* **Компоненты Quartus vs Эмуляция:** 
  В модулях надо проверять определён или нет дефайн `TESTMODE`. Если не определён, то для Dual Port RAM и Dual Clock FIFO надо использовать стандартные компоненты Quartus (IP Cores). Если определён — использовать эмуляцию (написать для них свой поведенческий код для симуляции, модули `dc_fifo` and `dp_ram`).
  
* **Правило сброса (Resets):** 
  Во всем проекте для всех тактовых доменов используются **исключительно синхронные сбросы с активным низким уровнем** (активный уровень — лог. `0`, суффикс `_n`), синхронизированные со своим тактовым доменом. Асинхронные сбросы на триггерах не допускаются во избежание проблем с нарушением временных ограничений (recovery/removal) на высоких частотах.

* **Централизованная синхронизация и сброс:**
  Все сигналы синхронного сброса (`o_<domain>_rst_n`) и сигналы синхронного запуска (`o_<domain>_sync`) для всех функциональных модулей проекта (таких как `ascan`, генераторы импульсов, ЦАП и т.д.) должны поступать **исключительно** с соответствующих выходов центрального модуля `rst_sync` (описан в п. 3.4). Прямое использование внешнего сигнала `rst_n` или несинхронизированных триггеров запуска внутри других модулей запрещено.

* **Правила именования портов (Naming Conventions):**
  * Входные порты всегда начинаются с префикса `i_` (например, `i_sys_sync`, `i_n_samples`).
  * Выходные порты всегда начинаются с префикса `o_` (например, `o_out_data`, `o_out_vld`).
  * **Исключение для клоков и сбросов:** Все тактовые частоты (clocks) и сбросы (resets) именуются **без префикса `i_`** (даже если они являются входными портами). Примеры: `sys_clk`, `adc_clk`, `rst_n`.
  * **Правило для сбросов (Resets):** Все сигналы сброса с активным низким уровнем (отрицательные) обязательно заканчиваются суффиксом `_n` (например, `rst_n`, `sys_rst_n`). Выходные синхронные сбросы из модулей синхронизации должны строго соответствовать шаблону `o_<domain>_rst_n` (например, `o_sys_rst_n`, `o_adc_rst_n`, `o_log_rst_n`, `o_dac_rst_n`, `o_hi_rst_n`).
  * Сигналы запуска (`sync`): перед словом `sync` обязательно должно идти имя домена.
    * Для входных сигналов запуска: `i_<domain>_sync` (например, `i_sys_sync`, `i_adc_sync`).
    * Для выходных сигналов запуска: `o_<domain>_sync` (например, `o_sys_sync`, `o_adc_sync`, `o_log_sync`, `o_dac_sync`, `o_hi_sync`).

## 3. Модули:

 ### 3.1 dc_fifo

 Модуля который будет эмулировать работу Dual Clock FIFO из Quartus для тестирования проекта с помощью icarus

 ### 3.2 dp_ram

 Модуля который будет эмулировать работу Dual Port RAM из Quartus для тестирования проекта с помощью icarus

 ### 3.3 Модуль накопления и выдачи данных от АЦП ascan
 название модуля - ascan
 тактовая частота для входных данных от АЦП - adc_clk
 запуск i_adc_sync (приходит в тактовом домене adc_clk)
 входные данные 12 бит знаковые
 Параметры:
 - [15:0] i_n_samples - количество отсчётов от АЦП
 - [7:0] i_accum - накопление (из скольки отсчётов надо получить одно значение на выходе)
 - [1:0] i_accum_type - тип накопления
 - [15:0] i_skip_ticks - задержка перед началом регистрации

 Выходная частота sys_clk
 Данные выдаются по потоковой шине
 - [31:0] o_out_data - выходные данные
 - o_out_vld - валидность выходных данных
 - i_out_rdy - готовность принимающей стороны
 - [15:0] o_out_size - количество данных готовых к передаче
 - o_data_ready - буфер заполнин и готов к передачи

 Данные от АЦП поступают на вход вспомогательного модуля который производит накопление, оттуда ни поступа на вход следующегго вспомогательного модуля,
 который перепаковывает их в 32-х битные слова, далее они в основном модуле накапливаются в двухпортовой памяти (память представляет из себя двойной буфер "бабочка") 
 и по готовности передаются по выходной потоковой шине

 ### 3.4 Модуль генерации синхронных сбросов и запусков rst_sync
 Название модуля — `rst_sync`. Путь к файлу: `src/rst_sync.v`.
 Модуль предназначен для генерации единой структуры синхронных сбросов и синхронных импульсов запуска (триггеров сбора) для всех тактовых доменов проекта. Все остальные модули проекта обязаны использовать сигналы сброса и запуска, приходящие только от этого модуля.

 **Внутренняя архитектура и алгоритм работы (согласно `src/rst_sync.v`):**
 1. **Генерация синхронных сбросов (`rst_gen`):**
    * Для каждого из 5 тактовых доменов реализован индивидуальный сдвиговый регистр инициализации шириной 8 бит.
    * При переходе внешнего асинхронного сигнала `rst_n` в низкий уровень регистры сбрасываются в `0`.
    * При освобождении `rst_n` регистры последовательно заполняются единицами на каждом такте своего домена. 
    * Выходной сигнал синхронного сброса `o_<domain>_rst_n` снимается (переходит в лог. `1`) строго синхронно с соответствующим клоком после удержания в активном низком состоянии не менее 8 тактов, что гарантирует корректную инициализацию всех IP-ядер и триггеров в проекте.
 2. **Выделение фронта запуска в системном домене:**
    * Входной сигнал запуска `i_sys_sync` (синхронный с `sys_clk`) подается на триггерный детектор переднего фронта.
    * На выходе формируется одиночный импульс запуска `o_sys_sync` длительностью ровно в 1 такт `sys_clk`.
 3. **Перенос импульса запуска через тактовые домены (CDC Pulse Sync):**
    * Для исключения метастабильности при передаче импульса запуска из `sys_clk` в другие несинхронные домены (`adc_clk`, `log_clk`, `dac_clk`, `hi_clk`) используется схема `cdc_pulse_sync`.
    * Одиночный импульс в домене-источнике переключает состояние внутреннего триггера (Toggle-генератор).
    * Полученный уровень безопасно транслируется в целевой тактовый домен с помощью 3-ступенчатого синхронизатора на триггерах.
    * В целевом домене схема XOR-детектора сравнивает состояние выходов соседних ступеней синхронизатора, восстанавливая стабильный одиночный импульс запуска (`o_adc_sync`, `o_log_sync`, `o_dac_sync`, `o_hi_sync`) длительностью строго в 1 такт целевой частоты.

 **Входные порты:**
 * Сетка частот: `sys_clk`, `adc_clk`, `log_clk`, `dac_clk`, `hi_clk`
 * Вход глобального сброса (может быть асинхронным): `rst_n` (активный уровень — низкий)
 * Вход запуска системы (мастер-запуск, синхронен с `sys_clk`): `i_sys_sync`

 **Выходные порты:**
 * **Синхронные сбросы (активный уровень — низкий (лог. 0), длительность удержания не менее 8 тактов своего домена для надежной инициализации IP-ядер):**
   - `o_sys_rst_n` (синхронен `sys_clk`)
   - `o_adc_rst_n` (синхронен `adc_clk`)
   - `o_log_rst_n` (синхронен `log_clk`)
   - `o_dac_rst_n` (синхронен `dac_clk`)
   - `o_hi_rst_n`  (синхронен `hi_clk`)
 * **Синхронные импульсы запуска (длительность строго 1 такт соответствующей частоты при появлении переднего фронта `i_sys_sync`):**
   - `o_sys_sync` (синхронен `sys_clk`)
   - `o_adc_sync` (синхронен `adc_clk`)
   - `o_log_sync` (синхронен `log_clk`)
   - `o_dac_sync` (синхронен `dac_clk`)
   - `o_hi_sync`  (синхронен `hi_clk`)
