# Анализ проекта и понимание задачи

## 1. Общий контекст системы (УЗ Дефектоскоп)
Мы проектируем высокоскоростной цифровой тракт для ультразвукового дефектоскопа. Прибор содержит:
*   **16 каналов возбуждения (генераторов)** для отправки УЗ-импульсов.
*   **4 физических приемника**, каждый из которых через коммутаторы (мультиплексоры 4-в-1) может подключаться к одной из 4 групп входов (всего 16 входных каналов).
*   **Различные аналоговые тракты:** один идет через VGA (усилитель с переменным коэффициентом усиления, например, AD604) на быстрый АЦП-1, второй — через логарифмический усилитель на АЦП-2.

Для синхронизации всех узлов используется сложная сетка частот (5 доменов):
*   `sys_clk` (80 МГц) — системная логика, общение с хостом, чтение готовых данных.
*   `adc_clk` (65 МГц) — захват данных с быстрого АЦП-1 (после VGA) и их первичная обработка (наш домен).
*   `log_clk` (25 МГц) — захват данных с логарифмического АЦП-2.
*   `dac_clk` (50 МГц) — управление ЦАП для формирования сигналов ВАРУ (временной регулировки чувствительности).
*   `hi_clk` (250 МГц) — высокоточный домен для формирования зондирующих импульсов с субтактовым разрешением (шаг 4 нс).

## 2. Понимание задачи: Разработка модуля `ascan`

Модуль `ascan` — это цифровой процессор сигналов А-скана (A-scan), который производит первичную фильтрацию/прореживание, накопление данных с быстрого АЦП и безопасный перенос этих данных в системный домен для дальнейшей отправки.

### А. Входные параметры и управление (в домене `adc_clk`):
*   **Запуск процесса:** Импульс `i_adc_sync` сигнализирует о начале цикла зондирования (синхронен `adc_clk`).
*   **Задержка регистрации (`i_skip_ticks`):** Модуль должен выждать заданное количество тактов `adc_clk` после `i_adc_sync` перед тем, как начать записывать и обрабатывать данные АЦП. Это нужно для исключения зоны зондирующего импульса (мертвой зоны) или выбора конкретного окна интереса.
*   **Длина записи (`i_n_samples`):** Количество исходных отсчетов АЦП, которые необходимо обработать.
*   **Коэффициент накопления/децимации (`i_accum`):** Задает размер группы отсчетов (например, каждые $N$ отсчетов объединяются в один выходной результат).
*   **Режим накопления (`i_accum_type`):** Определяет математическую операцию над группой из `i_accum` отсчетов:
    *   *Пиковый детектор (максимум)* — выделение огибающей (детектирование пиков). Благодаря предварительному взятию абсолютного значения, этот режим находит максимальную амплитуду сигнала независимо от его исходной полярности.
    *   *Интегрирование / Среднее арифметическое* — для фильтрации шумов. При этом режиме накопленная сумма выпрямленных значений делится на `i_accum` для возвращения к исходному масштабу.
    *   *Обычная децимация (выборка каждого N-го отсчета)*.

### Б. Внутренняя архитектура модуля (3 логических блока):
1.  **Вспомогательный модуль накопления (Sub-module 1: Accumulator):**
    *   Работает на частоте `adc_clk`.
    *   Принимает 12-битные знаковые данные от АЦП (`signed [11:0]`).
    *   **Предварительная обработка (Детектирование):** Первым шагом модуль вычисляет **абсолютное значение (модуль)** от каждого входного отсчета АЦП (`abs_data = |i_adc_data|`). Это необходимо для выпрямления высокочастотного заполнения ультразвукового импульса (RF) и получения его огибающей. 12-битное знаковое число преобразуется в 12-битное положительное (с ограничением/насыщением для предотвращения переполнения при взятии модуля от минимального отрицательного значения $-2048$).
    *   Реализует счетчик задержки (`i_skip_ticks`), счетчик обработанных точек (`i_n_samples`) и логику накопления выпрямленных данных в группы размером `i_accum` в соответствии с режимом `i_accum_type`.
    *   **Масштабирование:** На выходе модуля накопления данные всегда остаются **12-битными** (`[11:0]`). При использовании режима среднего арифметического накопленная сумма делится на коэффициент накопления `i_accum`, что возвращает результат в исходный 12-битный диапазон и предотвращает разрастание разрядности.
2.  **Модуль упаковщика (Sub-module 2: Packer):**
    *   Принимает накопленные и отнормированные 12-битные беззнаковые данные от модуля накопления по потоковому протоколу.
    *   **Честная плотная упаковка (Bit-Packing):** Для максимальной экономии памяти 12-битные отсчеты упаковываются в 32-битные слова без добавления пустых (выравнивающих) бит. Каждые **8 последовательных отсчетов** (S0..S7) по 12 бит упаковываются ровно в **3 слова** (W0..W2) по 32 бита ($8 \times 12 = 96$ бит):
        *   **Слово 0 (W0):**
            *   `W0[11:0]`   = Отсчет 0 (`S0[11:0]`)
            *   `W0[23:12]`  = Отсчет 1 (`S1[11:0]`)
            *   `W0[31:24]`  = Отсчет 2 (младшие 8 бит: `S2[7:0]`)
        *   **Слово 1 (W1):**
            *   `W1[3:0]`    = Отсчет 2 (старшие 4 бита: `S2[11:8]`)
            *   `W1[15:4]`   = Отсчет 3 (`S3[11:0]`)
            *   `W1[27:16]`  = Отсчет 4 (`S4[11:0]`)
            *   `W1[31:28]`  = Отсчет 5 (младшие 4 бита: `S5[3:0]`)
        *   **Слово 2 (W2):**
            *   `W2[7:0]`    = Отсчет 5 (старшие 8 бит: `S5[11:4]`)
            *   `W2[19:8]`   = Отсчет 6 (`S6[11:0]`)
            *   `W2[31:20]`  = Отсчет 7 (`S7[11:0]`)
    *   **Выравнивание конца кадра:** Если общее количество накопленных отсчетов не кратно 8, упаковщик заполняет оставшиеся свободные биты последнего 32-битного слова нулями, обеспечивая корректное завершение транзакции записи в память.
3.  **Двухпортовая буферная память "Бабочка" / Ping-Pong Buffer (Main Module):**
    *   Реализует безопасный переход между тактовыми доменами (CDC: Clock Domain Crossing) из `adc_clk` (65 МГц) в `sys_clk` (80 МГц).
    *   Состоит из двух независимых банков памяти RAM (двойной буфер). Пока в один буфер пишутся новые упакованные 32-битные данные от АЦП на частоте `adc_clk`, из второго буфера система вычитывает ранее накопленные данные на частоте `sys_clk`. Затем они меняются местами.
    *   Это гарантирует непрерывность сбора данных без потери темпа АЦП.

### В. Выходной интерфейс (в домене `sys_clk`):
*   Данные выдаются по стандартному потоковому интерфейсу (по типу AXI-Stream):
    *   `o_out_data [31:0]` — шина данных (содержит плотно упакованные 12-битные отсчеты).
    *   `o_out_vld` — сигнал валидности данных со стороны модуля.
    *   `i_out_rdy` — готовность приемника (хоста/процессора) принимать данные (Flow Control).
*   Вспомогательные статус-сигналы:
    *   `o_out_size [15:0]` — точный размер кадра (в 32-битных словах), готового к считыванию из текущего буфера (рассчитывается с учетом плотной упаковки).
    *   `o_data_ready` — флаг готовности буфера (сигнализирует, что один из полубуферов "бабочки" полностью заполнен и готов к чтению).

### Резюме по реализации:
Для реализации этого модуля потребуется написать:
1.  **FSM (конечный автомат) домена АЦП** для отработки задержки (`skip`), накопления (`accum`) и управления записью в RAM.
2.  **Логику выпрямления и накопления** (взятие абсолютного значения, поддержка разных режимов `i_accum_type` и деление/масштабирование до 12 бит).
3.  **Упаковщик в 32 бита** (честная плотная упаковка 12-битных данных по схеме 8-в-3, дополнение нулями на конце кадра).
4.  **Двухпортовую True Dual Port RAM** (или Simple Dual Port с раздельными клоками) и **FSM переключения буферов** (Ping-Pong) с безопасной передачей сигналов готовности через CDC-синхронизаторы.
5.  **FSM домена чтения (System Domain)** для выгрузки данных по интерфейсу Ready/Valid.
