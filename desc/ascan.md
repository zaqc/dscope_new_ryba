# Анализ проекта и понимание задачи

## 1. Общий контекст системы (УЗ Дефектоскоп)
Мы проектируем высокоскоростной цифровой тракт для ультразвукового дефектоскопа. Прибор содержит:
*   **16 каналов возбуждения (генераторов)** для отправки УЗ-импульсов.
*   **4 физических приемника**, каждый из которых через коммутаторы (мультиплексоры 4-в-1) может подключаться к одной из 4 групп входов (всего 16 входных каналов).
*   **Различные аналоговые тракты:** один идет через VGA (усилитель с переменным коэффициентом усиления, например, AD604) на быстрый АЦП-1, второй — через логарифмический усилитель на АЦП-2.

Для синхронизации всех узлов используется сложная сетка частот (5 доменов):
*   `sys_clk` (80 МГц) — системная логика, общение с хостом, чтение готовых данных.
*   `adc_clk` (65 МГц) — захват данных с быстрого АЦП-1 (после VGA) и их первичная обработка (наш домен).
*   `log_clk` (25 МГц) — захват данных с логарифмического АЦП-2.
*   `dac_clk` (50 МГц) — управление ЦАП для формирования сигналов ВАРУ (временной регулировки чувствительности).
*   `hi_clk` (250 МГц) — высокоточный домен для формирования зондирующих импульсов с субтактовым разрешением (шаг 4 нс).

## 2. Понимание задачи: Разработка модуля `ascan` с поддержкой субзапусков

Модуль `ascan` — это цифровой процессор сигналов А-скана (A-scan), который производит первичную фильтрацию/прореживание, накопление данных с быстрого АЦП, плотную побитовую упаковку и безопасный перенос данных в системный домен через двойной буфер. 

Для реализации работы с входными коммутаторами 1:4 физических каналов модуль поддерживает **последовательную многократную регистрацию (до 4 субнакоплений)** в рамках одного кадра (одного физического буфера RAM) по цепочке субзапусков.

### А. Входные параметры и управление (в домене `adc_clk`):
*   **Параметры измерения (могут динамически обновляться между субзапусками):**
    *   `[15:0] i_n_samples` — количество исходных отсчетов АЦП, которые необходимо обработать в текущей итерации.
    *   `[7:0] i_accum` — коэффициент накопления/децимации.
    *   `[1:0] i_accum_type` — тип накопления (пик-детектор, среднее арифметическое, децимация).
    *   `[15:0] i_skip_ticks` — задержка регистрации (в тактах `adc_clk`) перед началом текущей записи.
*   **Сигналы синхронизации и последовательного управления:**
    *   `i_adc_sync` — глобальный мастер-запуск кадра. Сбрасывает внутренний адрес записи RAM в `0`, сбрасывает счетчики субнакоплений и запускает первую итерацию регистрации.
    *   `i_sub_sync` — сигнал субзапуска (субстарт). Подается внешней логикой после перекоммутации входных каналов и обновления параметров. Запускает очередную итерацию регистрации. Запись в текущий полубуфер RAM продолжается с адреса, на котором завершилось предыдущее субнакопление.
    *   `i_sub_last` — флаг последнего субнакопления. Подается совместно с `i_adc_sync` или `i_sub_sync`. Сигнализирует о том, что текущее субнакопление является финальным в серии, после чего буфер «бабочки» должен быть переключен.
    *   `o_sub_done` — выходной сигнал готовности текущего субнакопления. Сигнализирует управляющему автомату о том, что модуль `ascan` завершил регистрацию текущей порции данных, упаковал ее, записал в RAM и готов к переключению коммутатора и приему импульса `i_sub_sync`.

### Б. Внутренняя архитектура модуля (3 логических блока):
1.  **Вспомогательный модуль накопления (Sub-module 1: Accumulator):**
    *   Работает на частоте `adc_clk`.
    *   Принимает 12-битные знаковые данные от АЦП (`signed [11:0]`).
    *   **Предварительная обработка (Детектирование):** Первым шагом модуль вычисляет **абсолютное значение (модуль)** от каждого входного отсчета АЦП (`abs_data = |i_adc_data|`). Это необходимо для получения огибающей. 12-битный знаковый сигнал преобразуется в 12-битный положительный (с ограничением/насыщением на уровне $+2047$ для предотвращения переполнения при модуле от значения $-2048$).
    *   Отрабатывает задержку `i_skip_ticks` после каждого `i_adc_sync` или `i_sub_sync`.
    *   Производит накопление/децимацию группы размером `i_accum` согласно выбранному `i_accum_type`.
    *   **Масштабирование:** На выходе данные всегда остаются **12-битными** (`[11:0]`). В режиме среднего арифметического сумма делится на `i_accum` для возвращения в исходный масштаб.
2.  **Модуль упаковщика (Sub-module 2: Packer):**
    *   Принимает 12-битные данные от накопителя по потоковому протоколу.
    *   **Честная плотная упаковка (Bit-Packing):** Упаковывает каждые 8 последовательных отсчетов (S0..S7) по 12 бит ровно в 3 слова по 32 бита ($8 \times 12 = 96$ бит):
        *   **Слово 0 (W0):** `W0[11:0] = S0`, `W0[23:12] = S1`, `W0[31:24] = S2[7:0]`
        *   **Слово 1 (W1):** `W1[3:0] = S2[11:8]`, `W1[15:4] = S3`, `W1[27:16] = S4`, `W1[31:28] = S5[3:0]`
        *   **Слово 2 (W2):** `W2[7:0] = S5[11:4]`, `W2[19:8] = S6`, `W2[31:20] = S7`
    *   **Выравнивание субнакоплений (Flush):** По окончании каждого субнакопления (сигнализируется завершением пачки данных) упаковщик делает принудительный сброс (Flush): дополняет текущее незавершенное 32-битное слово нулями и записывает его в RAM. Благодаря этому следующее субнакопление гарантированно начнется со строго выровненного адреса (с нового 32-битного слова).
3.  **Двухпортовая буферная память "Бабочка" / Ping-Pong Buffer (Main Module):**
    *   Реализует безопасный переход между тактовыми доменами (CDC) из `adc_clk` (65 МГц) в `sys_clk` (80 МГц).
    *   Пока в активный банк памяти последовательно записываются результаты субнакоплений (от 1 до 4 итераций) на частоте `adc_clk`, из второго банка система вычитывает ранее сформированный полный кадр на частоте `sys_clk`.
    *   Переключение банков ("бабочки") происходит только тогда, когда завершается субнакопление, запущенное с активным флагом `i_sub_last`.
    *   В момент переключения буферов значения размеров всех выполненных субнакоплений фиксируются и безопасно передаются в системный домен.

### В. Выходной интерфейс (в домене `sys_clk`):
*   Данные выдаются по стандартному потоковому интерфейсу (Ready/Valid):
    *   `o_out_data [31:0]` — шина данных (содержит последовательно расположенные блоки субнакоплений, каждый из которых выровнен по границе 32-битного слова).
    *   `o_out_vld` — сигнал валидности выходных данных.
    *   `i_out_rdy` — готовность приемника принимать данные.
*   **Статусные сигналы размеров (индивидуально для каждого субнакопления):**
    *   `o_out_size0 [15:0]` — размер 1-го субнакопления в 32-битных словах.
    *   `o_out_size1 [15:0]` — размер 2-го субнакопления в 32-битных словах (0, если не использовалось).
    *   `o_out_size2 [15:0]` — размер 3-го субнакопления в 32-битных словах (0, если не использовалось).
    *   `o_out_size3 [15:0]` — размер 4-го субнакопления в 32-битных словах (0, если не использовалось).
    *   `o_data_ready` — флаг готовности всего кадра (поднимается, когда вся серия субнакоплений завершена и буфер готов к считыванию).

### Резюме по реализации:
Для реализации этого модуля потребуется написать:
1.  **FSM (конечный автомат) домена АЦП** для управления последовательностью субнакоплений: отработка задержек, генерация `o_sub_done` для хоста, ожидание `i_sub_sync`, отслеживание признака `i_sub_last`.
2.  **Логику выпрямления и накопления** (взятие абсолютного значения, поддержка режимов `i_accum_type` и масштабирование обратно к 12 битам).
3.  **Упаковщик в 32 бита** (честная плотная упаковка 12-битных данных 8-в-3, функция принудительного выравнивания/дополнения нулями в конце каждого субнакопления).
4.  **Счетчик адреса RAM**, который сбрасывается в `0` только по `i_adc_sync`, инкрементируется при записи упакованных слов и сохраняет свое значение между субзапусками. Также необходимы регистры для фиксации размеров каждого из субнакоплений.
5.  **Двухпортовую RAM** и **FSM переключения буферов** (Ping-Pong) с безопасной CDC-передачей флага готовности `o_data_ready` и четырех регистров размеров в домен `sys_clk`.
6.  **FSM домена чтения (System Domain)** для последовательного чтения данных из RAM по интерфейсу Ready/Valid.
