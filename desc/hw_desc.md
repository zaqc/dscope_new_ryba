# Сетевое взаимодействие и ТЗ на тестовое приложение верхнего уровня (ПО ВУ)

Данный документ описывает протокол сетевого взаимодействия между 16-канальным ультразвуковым дефектоскопом и программным обеспечением верхнего уровня (ПО ВУ), а также содержит техническое задание на создание тестового приложения для калибровки, настройки параметров и визуализации результатов зондирования.

---

## 1. Архитектура сетевого взаимодействия

Устройство и ПО ВУ обмениваются данными по Ethernet через стек протоколов UDP/IP. 

```
  +-------------------------------------------------+
  |                  ПО ВУ (Хост)                   |
  |                                                 |
  |  [Отправка команд]       [Прием данных кадра]   |
  +--------+--------------------------^-------------+
           |                          |
           | UDP Unicast              | UDP Multicast
           | 192.168.11.11:2179       | 224.77.236.6:8569 (from port 20818)
           v                          |
  +-----------------------------------+-------------+
  |              16-канальный дефектоскоп           |
  |                                                 |
  | IP: 192.168.11.11                               |
  +-------------------------------------------------+
```

* **Сетевые параметры дефектоскопа (по умолчанию):**
  * Статический IP-адрес: `192.168.11.11`
  * Порт для приема команд: `2179`
* **Сетевые параметры трансляции данных:**
  * IP-адрес мультикаст-группы: `224.77.236.6`
  * Порт отправки (на стороне прибора): `20818`
  * Порт приема (на стороне хоста): `8569`

---

## 2. Формат команд управления (Control Commands)

Управление прибором осуществляется отправкой UDP-пакетов фиксированной длины **12 байт** (3 слова по 32 бита). Все многобайтовые поля передаются в порядке байт **Big-Endian** (сетевой порядок байт).

### Структура UDP-пакета команды:

| Смещение (байт) | Размер (байт) | Имя поля | Описание |
| :---: | :---: | :--- | :--- |
| 0 | 4 | `magic_word` | Маркер пакета команды. Константа `0xAA554453` (символы "DS") |
| 4 | 4 | `cmd_addr` | 32-битный адрес регистра/параметра ПЛИС (согласно карте адресов) |
| 8 | 4 | `cmd_data` | 32-битное значение параметра для записи |

### Разметка поля `cmd_addr` [31:0]:

Координатная сетка параметров адресует **16 независимых конфигураций** (4 физических канала $\times$ 4 виртуальных цикла/шага субзапуска):

| Биты | Размер | Описание |
| :--- | :---: | :--- |
| `[31:24]` | 8 бит | **ID модуля:**<br>• `0x01` — модуль приемника `ascan`<br>• `0x02` — модуль генератора `pulse`<br>• `0x03` — модуль ВРЧ/усиления `tune` |
| `[23:16]` | 8 бит | **ID параметра** внутри выбранного модуля (см. таблицу параметров ниже) |
| `[15:14]` | 2 бита | **Индекс физического канала** (`0..3`) |
| `[13:12]` | 2 бита | **Индекс виртуального цикла / субзапуска** (`0..3`) |
| `[11:0]`  | 12 бит | Зарезервировано (заполняется нулями `0x000`) |

### Таблица поддерживаемых параметров для записи:

| ID Модуля | ID Параметра | Имя параметра | Формат данных в `cmd_data` | Описание |
| :---: | :---: | :--- | :--- | :--- |
| **0x01** | **0x01** | `n_samples` | `[15:0]` | Количество регистрируемых отсчетов АЦП |
| | **0x02** | `accum` | `[7:0]` | Коэффициент накопления/децимации ($M = 1 \dots 256$) |
| | **0x03** | `accum_type`| `[1:0]` | Режим: `00`-Пиковый, `01`-Среднее, `10`-Децимация |
| | **0x04** | `drop_ticks` | `[15:0]` | Задержка запуска регистрации в тактах `adc_clk` (65 МГц) |
| | **0x05** | `pep_idx` | `[1:0]` | Выбор физического ПЭП на коммутаторе 1:4 (`0..3`) |
| **0x02** | **0x01** | `charge` | `[15:0]` | Время заряда катушки индуктивности в тактах `hi_clk` (4 нс) |
| | **0x02** | `transfer` | `[15:0]` | Время переноса энергии в емкость в тактах `hi_clk` (4 нс) |
| | **0x03** | `strike` | `[7:0]` | Время разряда конденсатора на датчик в тактах `hi_clk` (4 нс) |
| | **0x04** | `gen_mask` | `[3:0]` | Маска подключения 4 физических генераторов к каналу |
| **0x03** | **0x01** | `start_amp` | `[10:0]` | Начальное общее усиление (0..2047) |
| | **0x02** | `amp_one` | `[31:0]` | Шаг приращения на 1-м сегменте (умножен на $2^{16}$) |
| | **0x03** | `amp_two` | `[31:0]` | Шаг приращения на 2-м сегменте (умножен на $2^{16}$) |
| | **0x04** | `vrc_len` | `[15:0]` | Длина первого сегмента (точка перегиба) в тактах `dac_clk` |
| | **0x05** | `dac_limits`| `[21:0]` | Упакованные лимиты ЦАП:<br>`[9:0]` dac_min, `[19:10]` dac_max, `[21:20]` tune_mode |
| | **0x06** | `log_offset`| `[9:0]` | Смещение логарифмического тракта AD8310 |
| | **0x07** | `drop_ticks`| `[15:0]` | Задержка начала нарастания кривой ВРЧ в тактах `dac_clk` |

*Пример сборки `cmd_addr`:*
Записать параметр `n_samples` (ID `0x01`) для модуля `ascan` (ID `0x01`) физического канала 2, виртуального шага 3.
* ID Модуля = `0x01` (биты 31:24)
* ID Параметра = `0x01` (биты 23:16)
* Физический канал = `2` (`2'b10`, биты 15:14)
* Виртуальный цикл = `3` (`2'b11`, биты 13:12)
* Результат: `cmd_addr = 0x0101B000`

## 3. Формат выходных данных (Output Frame Format)

Данные считываются из Ping-Pong буфера ("бабочки") модуля `ascan_hub` в системном домене `sys_clk` и транслируются в Ethernet в виде единого непрерывного кадра. 

### Структура Ethernet-пакета данных:

Каждый полученный мультикаст-пакет содержит **Системный заголовок**, **Заголовок метаданных кадра** и **Массив полезных данных**.

```
+-----------------------------------------------------------------------+
|  Системный Ethernet-заголовок пакета (16 байт)                        |
|  - Magic Word (0x4153434E) | Номер кадра | Полный размер | Флаги       |
+-----------------------------------------------------------------------+
|  Заголовок метаданных кадра (32 байта / 8 слов по 32 бита)             |
|  - Размеры блоков данных для каждого шага s [0..3] и канала c [0..3]  |
+-----------------------------------------------------------------------+
|  Полезная нагрузка (Payload)                                          |
|  - Последовательные блоки упакованных отсчетов АЦП.                   |
|  - Порядок выдачи: Step 0 (Ch0, Ch1, Ch2, Ch3) -> Step 1 (Ch0...)     |
+-----------------------------------------------------------------------+
```

### 3.1. Системный заголовок пакета (16 байт)

| Смещение (байт) | Тип данных | Имя поля | Описание |
| :---: | :---: | :--- | :--- |
| 0 | `uint32` | `sys_magic` | Константа синхронизации: `0x4153434E` ("ASCN") |
| 4 | `uint32` | `frame_counter`| Инкрементальный счетчик отправленных кадров ПЛИС |
| 8 | `uint32` | `total_words` | Общий размер пакета в 32-битных словах (Header + Payload) |
| 12 | `uint32` | `sys_flags` | Флаги состояния (например, переполнение буфера FIFO ПЛИС) |

### 3.2. Заголовок метаданных кадра (32 байта)

Данный блок содержит длины записанных массивов данных (в 32-битных словах) для всех комбинаций шагов и каналов.

| Номер слова | Биты [31:16] | Биты [15:0] | Описание |
| :---: | :--- | :--- | :--- |
| **0** | `ch1_size_step0` | `ch0_size_step0` | Размеры для **Шага 0** (Канал 1 / Канал 0) |
| **1** | `ch3_size_step0` | `ch2_size_step0` | Размеры для **Шага 0** (Канал 3 / Канал 2) |
| **2** | `ch1_size_step1` | `ch0_size_step1` | Размеры для **Шага 1** (Канал 1 / Канал 0) |
| **3** | `ch3_size_step1` | `ch2_size_step1` | Размеры для **Шага 1** (Канал 3 / Канал 2) |
| **4** | `ch1_size_step2` | `ch0_size_step2` | Размеры для **Шага 2** (Канал 1 / Канал 0) |
| **5** | `ch3_size_step2` | `ch2_size_step2` | Размеры для **Шага 2** (Канал 3 / Канал 2) |
| **6** | `ch1_size_step3` | `ch0_size_step3` | Размеры для **Шага 3** (Канал 1 / Канал 0) |
| **7** | `ch3_size_step3` | `ch2_size_step3` | Размеры для **Шага 3** (Канал 3 / Канал 2) |

*Примечание:* Размер блока данных измеряется в **32-битных словах**. Если для конкретного шага и канала измерение не проводилось, размер равен `0`.

### 3.3. Полезная нагрузка (Payload)

Полезная нагрузка содержит последовательные массивы данных. Обход осуществляется в порядке: **внешний цикл — Шаги (0..3), внутренний цикл — Каналы (0..3)**.

```
Массив Payload = [ Данные(S0, Ch0) ] + [ Данные(S0, Ch1) ] + ... + [ Данные(S3, Ch3) ]
```
Если размер блока для пары `(Step S, Channel C)` равен 0, то этот блок просто отсутствует в потоке данных (сразу идет следующий ненулевой блок).

## 4. Математический алгоритм декомпрессии данных (Unpacking)

Внутри каждого блока данные оцифровки упакованы по принципу **8 отсчетов АЦП (по 12 бит каждый) в 3 слова по 32 бита** (всего 96 бит). Входные отсчеты представляют собой положительную огибающую сигнала (модуль знакового отсчета АЦП), поэтому их физический диапазон равен `0..2047` (11 значащих бит + 1 бит защиты от переполнения).

Для восстановления исходных отсчетов $S_0 \dots S_7$ из триады 32-битных слов $W_0, W_1, W_2$ необходимо применить следующую схему побитового сдвига и маскирования (с маской `0xFFF`):

```
Слово 0 (W0): [ S2[7:0] (8b) ] [    S1 (12b)    ] [    S0 (12b)    ]
Слово 1 (W1): [ S5[3:0] (4b) ] [    S4 (12b)    ] [    S3 (12b)    ] [ S2[11:8] (4b) ]
Слово 2 (W2): [    S7 (12b)    ] [    S6 (12b)    ] [ S5[11:4] (8b) ]
```

### Формулы распаковки:

$$S_0 = W_0 \ \& \ \text{0xFFF}$$

$$S_1 = (W_0 \gg 12) \ \& \ \text{0xFFF}$$

$$S_2 = ((W_1 \ \& \ \text{0x00F}) \ll 8) \ | \ ((W_0 \gg 24) \ \& \ \text{0x0FF})$$

$$S_3 = (W_1 \gg 4) \ \& \ \text{0xFFF}$$

$$S_4 = (W_1 \gg 16) \ \& \ \text{0xFFF}$$

$$S_5 = (((W_2 \ \& \ \text{0x0FF}) \ll 4) \ | \ ((W_1 \gg 28) \ \& \ \text{0x00F}))$$

$$S_6 = (W_2 \gg 8) \ \& \ \text{0xFFF}$$

$$S_7 = (W_2 \gg 20) \ \& \ \text{0xFFF}$$

## 5. Техническое задание на разработку ПО верхнего уровня

### 5.1. Общие требования
1. **Язык разработки:** Python (с библиотеками `PyQt5`/`PySide2`, `pyqtgraph`, `numpy`) или C++ (с использованием фреймворка `Qt`).
2. **Тип приложения:** Кроссплатформенное десктопное приложение (Windows 10/11, Linux Ubuntu 20.04+).
3. **Производительность:** Обработка и плавная отрисовка до 50 полных кадров в секунду при максимальном объеме выборки (нагрузка на CPU < 15%).

### 5.2. Функциональные требования
1. **Панель конфигурирования (Сетка 4x4):**
   * Интерактивная таблица для настройки параметров. Должен поддерживаться выбор конкретного узла «Канал / Шаг» для индивидуальной настройки либо групповая запись параметров на все каналы сразу.
   * Поля ввода с валидацией диапазонов в соответствии с физическими ограничениями регистров ПЛИС.
   * Кнопка «Применить конфигурацию», которая формирует и отправляет пачку UDP команд на дефектоскоп.

2. **Интерактивный редактор кривой ВРЧ (TGC Editor):**
   * Графический холст, на котором отображается строящаяся кусочно-линейная кривая ВРЧ в координатах: Ось X — Время (такты/микросекунды), Ось Y — Код ЦАП / Относительное усиление в дБ.
   * Возможность перетаскивания "мышечным" вводом опорных точек:
     * Точки начала нарастания (`drop_ticks`).
     * Точки перегиба (`vrc_len`).
     * Уровней наклона (`amp_one`, `amp_two`).
   * Автоматический пересчет перетаскиваний в целочисленные коэффициенты с фиксированной точкой (формат Q16.16 для `amp_one/two`) и немедленная отправка в прибор.

3. **Блок визуализации А-сканов:**
   * Многоканальный осциллографический дисплей на базе `pyqtgraph` (для быстрой отрисовки).
   * Выбор режима отображения:
     * *Single Channel/Step Mode:* Отображение подробного сигнала от одной выбранной пары `(Channel, Step)`.
     * *Grid Mode:* Сетка из 4 графиков (физические каналы 0..3), на каждом из которых наложены друг на друга сигналы со всех активных шагов субзапуска.
   * Автоматическое масштабирование по осям X (время пробега) и Y (амплитуда 0..2047).
   * Наличие измерительных маркеров (курсоров) для измерения расстояния между эхо-сигналами (с пересчетом времени в миллиметры пути на основе заданной скорости ультразвука).

4. **Система логирования и экспорта:**
   * Запись сырого сетевого UDP-потока в бинарные файлы `.raw` для последующего воспроизведения (режим эмуляции устройства на хосте).
   * Возможность экспорта текущего кадра со всеми метаданными в формат CSV или JSON.
