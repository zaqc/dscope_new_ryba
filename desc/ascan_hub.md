# Модуль Ascan Hub (Концентратор приемных каналов А-скана)

Модуль `ascan_hub` является центральным связующим звеном приемного тракта ультразвукового дефектоскопа. Он объединяет 4 независимых параллельных физических канала накопления и предварительной обработки данных АЦП (модули `ascan`), выполняет упаковку их метаданных (размеров блоков) и данных в единый структурированный поток и выдает его через один общий высокопроизводительный потоковый интерфейс в системный домен.

Применение параметризованного генератора `generate` на основе внутренних массивов проводников совместно с управляющим автоматом упаковки (Packetizer FSM) полностью устраняет избыточность описания логики, защищает от ошибок ручной коммутации портов и упрощает интеграцию с системным DMA/FIFO.

---

## 1. Архитектура и системная интеграция

Модуль `ascan_hub` выполняет роль высокопроизводительного концентратора и упаковщика данных, работающего на стыке двух ключевых тактовых доменов: домена оцифровки (`adc_clk`) и системного домена (`sys_clk`).

```
                     +-------------------------------------------------------+
                     |                       ascan_hub                       |
                     |                                                       |
    adc_clk -------->| +------------+  +------------+  +------------+        |
   adc_rst_n ------->| |  ascan 0   |  |  ascan 1   |  |  ascan 2   | ...    |
    i_adc_sync ----->| | (Channel 0)|  | (Channel 1)|  | (Channel 2)|        |
    i_sub_sync ----->| +------------+  +------------+  +------------+        |
    i_sub_last ----->|       |               |               |               |
                     |       v               v               v               |=====> o_adc_chan_ready [3:0]
                     | [  RAM Ping  ] [  RAM Ping  ] [  RAM Ping  ]          |       (к sequencer)
                     | [  RAM Pong  ] [  RAM Pong  ] [  RAM Pong  ]          |
                     |       |               |               |               |
                     |       +---------------+---------------+               |
                     |                       |                               |
                     |                       v                               |
                     |         +---------------------------+                 |
                     |         |  Packetizer FSM & Mux     |                 |
                     |         +---------------------------+                 |
                     |                       |                               |
    sys_clk -------->|                       v                               |
   sys_rst_n ------->|         ======[ o_out_data [31:0] ]===================|=====> Единый поток кадра
                     |         ======[ o_out_vld         ]===================|=====> (sys_clk)
                     |         <=====[ i_out_rdy         ]===================|
                     |         ======[ o_out_size [15:0] ]===================|=====> Полный размер пакета
                     |         ======[ o_data_ready      ]===================|=====> Пакет собран и стабилен
                     +-------------------------------------------------------+
```

### Основные функции концентратора:
1. **Параллельная фильтрация и накопление**: Модуль распределяет входные потоки данных от 4 физических АЦП (`i_adc_data_ch0..3`) по соответствующим конвейерам обработки.
2. **Координация многошаговых циклов зондирования**: 
   * По сигналам `i_sub_sync` (запуск шага) и `i_sub_last` (финальный шаг кадра) каждый канал независимо регистрирует данные для текущего активного шага $s \in [0..3]$.
   * По окончании каждого шага модули `ascan` выставляют индивидуальные флаги готовности, которые объединяются в 4-битную шину `o_adc_chan_ready` для автомата управления `sequencer`.
3. **Двухстраничная буферизация («Бабочка» / Ping-Pong)**: 
   * Внутри каждого канала `ascan` интегрирован двойной буфер на базе двухпортовой памяти (`dp_ram`).
   * В то время как один буфер (страница) осуществляет скоростную запись очередного кадра в домене `adc_clk`, второй буфер доступен для непрерывного чтения системным контроллером в домене `sys_clk`.
   * Переключение страниц происходит строго синхронно по завершении финального субцикла кадра (`i_sub_last` совместно с завершением сбора во всех каналах).
4. **Шагово-последовательная упаковка пакета (Step-First Packetizing)**: 
   * Внутренний конечный автомат (FSM) в домене `sys_clk` дожидается завершения сбора данных во всех каналах кадра.
   * Вычисляется суммарный размер пакета, включая фиксированный заголовок метаданных (8 слов по 32 бита).
   * По сигналу `o_data_ready` внешняя система начинает чтение. Модуль `ascan_hub` транслирует в поток сначала заголовок с размерами блоков, сгруппированными по шагам, а затем последовательно вычитывает и выдает данные по следующему принципу: **для каждого шага $s \in [0..3]$ поочередно выдаются блоки данных из каналов $0, 1, 2, 3$**.

## 2. Сетка тактовых доменов

Модуль осуществляет безопасный междоменный перенос данных и управляющих сигналов.

| Имя тактовой частоты | Частота | Описание домена | Сброс (от `rst_sync`) |
| :--- | :--- | :--- | :--- |
| **adc_clk** | 65 MHz | Домен оцифровки АЦП. Захват данных, прореживание, накопление и запись в RAM | `adc_rst_n` |
| **sys_clk** | 80 MHz | Системный домен. Чтение накопленных кадров, передача в FIFO/DMA | `sys_rst_n` |

## 3. Порты модуля

### Системные интерфейсы, тактирование и сбросы
| Имя порта | Направление | Разрядность | Описание |
| :--- | :---: | :---: | :--- |
| **adc_clk** | вход | 1 | Тактовая частота домена АЦП (65 МГц) |
| **adc_rst_n** | вход | 1 | Синхронный сброс домена `adc_clk` (активный лог. `0`) |
| **sys_clk** | вход | 1 | Тактовая частота системного домена (80 МГц) |
| **sys_rst_n** | вход | 1 | Синхронный сброс домена `sys_clk` (активный лог. `0`) |

### Управляющие и синхронизирующие интерфейсы (домен `adc_clk`)
| Имя порта | Направление | Разрядность | Описание |
| :--- | :---: | :---: | :--- |
| **i_adc_sync** | вход | 1 | Сигнал глобального запуска кадра сканирования (одиночный импульс) |
| **i_sub_sync** | вход | 1 | Импульс запуска очередного шага (субцикла) зондирования |
| **i_sub_last** | вход | 1 | Признак финального шага в текущем кадре сканирования |
| **o_adc_chan_ready** | выход | 4 | Объединенный вектор готовности каналов (бит `i` соответствует готовности `ascan[i]`) |

### Входные данные от физических АЦП (домен `adc_clk`, знаковые данные)
| Имя порта | Направление | Разрядность | Описание |
| :--- | :---: | :---: | :--- |
| **i_adc_data_ch0** | вход | 12 | Входные знаковые данные АЦП. Физический канал 0 |
| **i_adc_data_ch1** | вход | 12 | Входные знаковые данные АЦП. Физический канал 1 |
| **i_adc_data_ch2** | вход | 12 | Входные знаковые данные АЦП. Физический канал 2 |
| **i_adc_data_ch3** | вход | 12 | Входные знаковые данные АЦП. Физический канал 3 |

### Параметры конфигурации каналов (захватываются по триггерам запуска в домене `adc_clk`)
*Каждый из 4-х каналов (с префиксами `_ch0_` ... `_ch3_`) имеет идентичный набор настроек:*
| Шаблон имени порта | Направление | Разрядность | Описание |
| :--- | :---: | :---: | :--- |
| **i_ascan_chX_n_samples** | вход | 16 | Количество отсчетов АЦП, регистрируемых в одном шаге |
| **i_ascan_chX_accum** | вход | 8 | Коэффициент накопления (число объединяемых отсчетов) |
| **i_ascan_chX_accum_type** | вход | 2 | Режим накопления (00: пиковый детектор, 01: усреднение и т.д.) |
| **i_ascan_chX_skip_ticks** | вход | 16 | Задержка запуска регистрации (в тактах `adc_clk`) относительно `sub_sync` |

### Объединенный выходной потоковый интерфейс (домен `sys_clk`, потоковая шина)
| Имя порта | Направление | Разрядность | Описание |
| :--- | :---: | :---: | :--- |
| **o_out_data** | выход | 32 | Выходные данные (32-битное слово: сначала заголовок, затем полезная нагрузка) |
| **o_out_vld** | выход | 1 | Сигнал валидности данных (Ready/Valid Handshake) |
| **i_out_rdy** | вход | 1 | Готовность принимающей стороны (Ready/Valid Handshake) |
| **o_out_size** | выход | 16 | Полный размер пакета в 32-битных словах (включая заголовок и данные) |
| **o_data_ready** | выход | 1 | Флаг готовности полного кадра (стабилен в памяти и готов к передаче) |

## 4. Структура выходного пакета (Frame Format)

Пакет, передаваемый через шину `o_out_data`, имеет строгую структуру, где метаданные (размеры) и данные упорядочены последовательно по шагам зондирования (субзапускам):

| Номер 32-битного слова | Биты [31:16] | Биты [15:0] | Описание метаданных и данных |
| :---: | :--- | :--- | :--- |
| **0** | `ch1_out_size0` (Шаг 0, Канал 1) | `ch0_out_size0` (Шаг 0, Канал 0) | Размеры блоков для **Шага 0** |
| **1** | `ch3_out_size0` (Шаг 0, Канал 3) | `ch2_out_size0` (Шаг 0, Канал 2) | Размеры блоков для **Шага 0** |
| **2** | `ch1_out_size1` (Шаг 1, Канал 1) | `ch0_out_size1` (Шаг 1, Канал 0) | Размеры блоков для **Шага 1** |
| **3** | `ch3_out_size1` (Шаг 1, Канал 3) | `ch2_out_size1` (Шаг 1, Канал 2) | Размеры блоков для **Шага 1** |
| **4** | `ch1_out_size2` (Шаг 2, Канал 1) | `ch0_out_size2` (Шаг 2, Канал 0) | Размеры блоков для **Шага 2** |
| **5** | `ch3_out_size2` (Шаг 2, Канал 3) | `ch2_out_size2` (Шаг 2, Канал 2) | Размеры блоков для **Шага 2** |
| **6** | `ch1_out_size3` (Шаг 3, Канал 1) | `ch0_out_size3` (Шаг 3, Канал 0) | Размеры блоков для **Шага 3** |
| **7** | `ch3_out_size3` (Шаг 3, Канал 3) | `ch2_out_size3` (Шаг 3, Канал 2) | Размеры блоков для **Шага 3** |
| **8 .. (8 + $N_{s0,ch0}$ - 1)** | <center>Данные `ch0`, Шаг 0</center> | <center>Данные `ch0`, Шаг 0</center> | Буфер канала 0 для шага 0 |
| **...** | <center>Данные `ch1`, Шаг 0</center> | <center>Данные `ch1`, Шаг 0</center> | Буфер канала 1 для шага 0 |
| **...** | <center>Данные `ch2`, Шаг 0</center> | <center>Данные `ch2`, Шаг 0</center> | Буфер канала 2 для шага 0 |
| **...** | <center>Данные `ch3`, Шаг 0</center> | <center>Данные `ch3`, Шаг 0</center> | Буфер канала 3 для шага 0 |
| **...** | <center>Данные `ch0`, Шаг 1</center> | <center>Данные `ch0`, Шаг 1</center> | Буфер канала 0 для шага 1 |
| **...** | <center>Данные `ch1`, Шаг 1</center> | <center>Данные `ch1`, Шаг 1</center> | Буфер канала 1 для шага 1 |
| **...** | <center>Данные `ch3`, Шаг 3</center> | <center>Данные `ch3`, Шаг 3</center> | Последний блок данных (канал 3, шаг 3) |

*Примечание: $N_{sY,chX}$ — объем накопленных 32-битных слов в канале $X$ конкретно за шаг $Y$. Если в каком-то канале на определенном шаге сбор не производился или размер блока равен 0, этот сегмент данных автоматически пропускается, а в заголовке для него будут переданы нулевые размеры. Это гарантирует бесперебойность потока.*

## 5. Алгоритм работы и конечный автомат упаковщика (Packetizer FSM)

Для выдачи пакета используется управляющий автомат в домене `sys_clk`. Ниже описаны его основные состояния:

1. **IDLE (Ожидание)**:
   * Автомат находится в ожидании, пока все активные внутренние каналы `ascan` не завершат сбор кадра и не выставят свои внутренние флаги готовности.
   * Рассчитывается суммарный размер пакета: 
     $$o\_out\_size = 8 + \sum_{s=0}^{3}\sum_{c=0}^{3} size(s, c)$$
   * Выставляется флаг готовности кадра: `o_data_ready = 1`. При переходе принимающей стороны в готовность (`i_out_rdy == 1`) совершается переход в состояние передачи заголовка.

2. **SEND_HEADER (Передача метаданных)**:
   * Последовательно за 8 тактов (при условии `i_out_rdy == 1`) выдаются 32-битные слова заголовка.
   * Данные выдаются в порядке возрастания шагов: `Word 0` и `Word 1` содержат длины для шага 0, `Word 2` и `Word 3` — для шага 1 и т.д.

3. **SEND_PAYLOAD (Передача полезной нагрузки)**:
   * Конечный автомат выполняет обход данных по двухпроходной схеме: внешний счетчик перебирает шаги $s \in [0..3]$, внутренний — физические каналы $c \in [0..3]$.
   * Для каждого узла `(step s, channel c)` проверяется размер записанного блока `size(s, c)`.
   * Если `size(s, c) > 0`, автомат выставляет соответствующий адрес чтения во внутренний буфер `ascan[c]` для сегмента шага `s`, выдерживает латентность памяти и транслирует данные в `o_out_data` с удержанием `o_out_vld`.
   * Если `size(s, c) == 0`, автомат мгновенно переходит к следующему каналу/шагу, не тратя такты шины на пустые пакеты.

4. **FRAME_DONE (Завершение)**:
   * После отправки последнего слова последнего активного канала на шаге 3 сбрасывается флаг `o_data_ready`, автомат переходит в состояние `IDLE`, инициируя переключение Ping-Pong страниц для приема новых данных от АЦП.

## 6. Временные соотношения и CDC (междоменная передача данных)

* **Исключение метастабильности**: Атомарное переключение Ping-Pong страниц памяти гарантирует, что пока FSM в домене `sys_clk` вычитывает страницу `A`, новые данные от АЦП гарантированно записываются в страницу `B` в домене `adc_clk`.
* **Передача размеров блоков**: Сигналы размеров блоков (`out_size0..3`) стабилизируются в домене `adc_clk` до выставления флага готовности и передаются в домен `sys_clk` через защищенный квитированием (Handshake / Gray-code) интерфейс внутри каждого модуля `ascan`, что полностью исключает чтение некорректных размеров в FSM упаковщика.
