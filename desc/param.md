### Анализ проблемы гонки параметров (Race Condition) в многоканальной системе

При использовании одинарного буфера активных регистров возникает критическая уязвимость в непрерывном цикле измерений:

1. **Запуск цикла N (`i_sys_sync` №1):**
   * Параметры из теневых регистров копируются в активные регистры.
   * Начинается физическое измерение №1. Модуль `ascan` записывает оцифрованные данные в первый буфер двойного буфера («бабочки») — **Буфер 0**.
2. **Окончание измерения №1:**
   * Запись в **Буфер 0** завершена. Выставляется флаг готовности `o_data_ready = 1`.
   * Однако хост-система или передатчик заняты (сигнал готовности шины чтения `i_out_rdy` равен `0`). Данные ждут отправки.
3. **Запуск цикла N+1 (`i_sys_sync` №2):**
   * Приходит следующий импульс запуска. Начинается измерение №2. Данные АЦП пишутся во второй буфер — **Буфер 1**.
   * **Проблема:** Если активные регистры в `param` одиночные, то по этому синку они мгновенно перезаписываются новыми значениями (для измерения №2).
4. **Считывание данных измерения №1:**
   * Наконец, передатчик готов (`i_out_rdy = 1`) и начинает вычитывать **Буфер 0** (измерение №1).
   * Вместе с данными он вычитывает метаданные из модуля параметров.
   * **Итог:** Передатчик прикрепит к физическим данным измерения №1 новые параметры от измерения №2. Произойдет искажение метаданных (разъюстировка параметров физического кадра, включая настройки ВРЧ, маски генераторов и выбранные ПЭП).

---

### Решение: Двойной буфер активных регистров (Ping-Pong параметров)

Для полной изоляции параметров физического измерения от параметров считывания мы добавляем в модуль `param` **двухбанковую структуру активных регистров** (Bank 0 и Bank 1) и внутренний 1-битный указатель `r_wr_ptr`:

1. **Внутренний указатель записи `r_wr_ptr`:**
   * Инициализируется в `0`.
   * Инвертируется (переключается `0 -> 1 -> 0`) строго по каждому импульсу `i_sys_sync`.
2. **Запись параметров текущего измерения (Захват):**
   * По сигналу `i_sys_sync` параметры из теневых регистров копируются в банк, на который указывает `r_wr_ptr` (например, в `reg_bank[r_wr_ptr]`).
   * Из этого же банка (`r_wr_ptr`) берутся параметры для немедленной передачи в домены физического захвата (`adc_clk`, `dac_clk` и `hi_clk`) через CDC.
3. **Чтение параметров предыдущего измерения (Выдача результатов в пакет):**
   * Все выходные порты для формирования пакета результатов (`o_sys_ascan_*`, `o_sys_pulse_*` и `o_sys_tune_*`) подключаются к противоположному банку: `reg_bank[~r_wr_ptr]`.
   * Поскольку цикл чтения всегда отстает от цикла записи ровно на 1 шаг (пока мы записываем новые данные в Буфер 1, мы передаем ранее накопленные данные из Буфера 0), параметры для передачи гарантированно будут считаны из «замороженного» банка, который не изменится до следующего синка.

Это полностью разделяет контур реального времени (физические модули) и контур вывода данных (интерфейс чтения), вне зависимости от того, в каком тактовом домене работает целевой модуль.

### Обзор проектных решений для модуля `param`

1. **Архитектура теневых регистров (Shadow/Holding Registers):**
   Модуль использует двухступенчатую структуру хранения. Запись по шине команд (`i_cmd_addr`, `i_cmd_data`, `i_cmd_vld`) в тактовом домене `sys_clk` всегда происходит только во временные теневые (Holding) регистры. Всего существует **16 наборов параметров**, структурированных как координатная сетка: **4 физических канала $\times$ 4 виртуальных цикла (субзапуска)**.

2. **Двойная буферизация активных регистров (Ping-Pong) для защиты от гонки данных:**
   В непрерывном режиме работы дефектоскопа новое измерение запускается по `i_sys_sync`, а данные предыдущего измерения еще могут находиться в процессе передачи в хост. Чтобы исключить перезапись метаданных кадра, модуль `param` реализует двухбанковую структуру активных регистров:
   * **Указатель `r_wr_ptr`:** 1-битный регистр в домене `sys_clk`, который переключается по каждому импульсу `i_sys_sync`. Он определяет, в какой банк будут сохранены параметры для нового измерения.
   * **Банк 0 (`bank_0`) и Банк 1 (`bank_1`):** Два идентичных набора активных регистров, каждый хранит настройки для 4 физических каналов по 4 виртуальных цикла в каждом.
   * **Логика работы:**
     * По сигналу `i_sys_sync` параметры из теневых регистров переносятся в банк, указанный `r_wr_ptr`.
     * На основе динамического входного индекса виртуального цикла `i_sys_vch_sel` (диапазон `0..3`), переключающегося во время измерительного процесса, из текущего рабочего банка извлекаются параметры для 4 параллельных физических трактов.
     * Каждый физический тракт $P$ ($0..3$) мгновенно получает параметры, соответствующие логическому набору: `Физический канал P / Виртуальный цикл i_sys_vch_sel`.
     * Параметры физических каналов (`o_ascan_ch*` в домен `adc_clk`, `o_tune_ch*` в домен `dac_clk`, `o_pulse_ch*` в домен `hi_clk`, а также коммутационные выходы `o_ascan_ch*_pep_idx` и `o_pulse_ch*_gen_mask` в домен `sys_clk`) уходят в работу.
     * Доступ к параметрам для формирования пакета результатов (`o_sys_ascan_*`, `o_sys_pulse_*`, `o_sys_tune_*`) осуществляется по двухкомпонентному адресу: **`i_packet_phy_ch`** (номер физ. канала `0..3`) и **`i_packet_vch`** (номер субзапуска `0..3`). Данные считываются строго из противоположного («замороженного») банка `~r_wr_ptr`.

3. **Безопасная передача параметров через тактовые домены (CDC):**
   Параметры передаются в несинхронные домены (`adc_clk`, `dac_clk` и `hi_clk`) по классической схеме **синхронизации квазистатических шин по импульсу запуска**:
   * В домене-источнике (`sys_clk`) мультиплексированные данные 4-х физических каналов фиксируются в выбранном банке по сигналу `i_sys_sync`.
   * В целевых доменах данные защелкиваются в локальные выходные триггеры по сигналам запуска `i_adc_sync` / `i_dac_sync` / `i_hi_sync` (которые гарантированно приходят позже `i_sys_sync` из-за задержки CDC-цепочки в `rst_sync`). Поскольку к моменту прихода синков данные на шине из `sys_clk` уже стабильны, метастабильность исключена.

### module ascan - номер модуля 1
модуль накопления значений АЦП, входные параметры приходят в домене частоты adc_clk

 - [1:0] pep_idx - Индекс коммутатора (1:4) для приёмного тракта физического канала (работает в домене sys_clk)

### module pulse - номер модуля 2
модуль управляет зондирующим импульсом, входные параметры приходят в домене частоты hi_clk
 - [15:0] charge - Время зарядки катушки
 - [15:0] transfer - Время переноса энергии в конденсатор
 - [7:0] strike - Время на которое конденсатор разряжается на ПЭП
 - [3:0] gen_mask - Маска подключения физических генераторов для зондирования (работает в домене sys_clk)

---
### module tune - номер модуля 3
модуль регулировки усиления для двух усилителей AD603 регулируемых напряжением включенных последовательно, напряжение выдают 2 ЦАП DAC101S101.
есть ещё третий ЦАП - который регулирует смещение для второго тракта усиления который построен на логарифмическом усилителе AD8310
работает на частоте dac_clk (измерительные параметры фиксируются по i_dac_sync)

Входные параметры:
 - [10:0] i_start_amp - Начальное усиление
 - [15:0] i_drop_ticks - Количество тактов задержки перед нарастанием ВРЧ
 - [31:0] i_amp_one - приращение усиления до точки перегиба (значение инкремента для кривой ВРЧ умноженное на 2^N)
 - [31:0] i_amp_two - приращение усиления после точки перегиба
 - [15:0] i_vrc_len - точка перегиба в тактах dac_clk
 - [9:0] i_dac_min - минимальное значение выдаваемое на ЦАП (соответствует MIN дБ усиления, регулировка производится от этого значения)
 - [9:0] i_dac_max - максимальное значение выдаваемое на ЦАП (соответствует MAX дБ усиления регулировка производится до этого значения)
 - [1:0] i_tune_mode - режим регулировки последовательный/параллельный

 - [9:0] i_log_offset - смещение для логарифмического усилителя

Выходные шины:
 - [9:0] o_dac_one - значение для ЦАП 1
 - [9:0] o_dac_two - значение для ЦАП 2
 - [9:0] o_dac_offset - значение для ЦАП смещения

получает значения на частоте dac_clk, для выдачи по шине SPI используется частоту adc_clk

### module param

Модуль `param` принимает конфигурационные команды по входной шине в домене `sys_clk`, декодирует их, временно хранит в теневых регистрах, а по сигналу синхронизации копирует их в активную память. Модуль осуществляет переключение параметров 4-х физических трактов по значению динамического индекса текущего субзапуска `i_sys_vch_sel` и поддерживает пинг-понг буферизацию с двухкоординатным интерфейсом чтения метаданных для сборщика выходных пакетов.

```verilog
    output reg  [9:0]  o_sys_tune_dac_max,
    output reg  [1:0]  o_sys_tune_tune_mode,
    output reg  [9:0]  o_sys_tune_log_offset
);
```

#### Карта адресов команд (Address Mapping)

Команда декодируется по старшим байтам адреса: `i_cmd_addr[31:24]` — номер модуля, `i_cmd_addr[23:16]` — код параметра, а младшие биты указывают на конкретный физико-виртуальный узел.

| Поле адреса | Разрядность | Назначение |
| :--- | :--- | :--- |
| `i_cmd_addr[31:24]` | 8 бит | **ID модуля** (0x01 - ascan, 0x02 - pulse, 0x03 - tune) |
| `i_cmd_addr[23:16]` | 8 бит | **ID параметра** (см. таблицу ниже) |
| `i_cmd_addr[15:14]` | 2 бита | **Индекс физического канала** (0..3) |
| `i_cmd_addr[13:12]` | 2 бита | **Индекс виртуального цикла / субзапуска** (0..3) |
| `i_cmd_addr[11:0]`  | 12 бит | Зарезервировано (не используется, должно быть равен 0) |

**Таблица параметров:**

| Модуль (ID) | Параметр (ID) | Имя параметра | Описание и разрядность |
| :--- | :--- | :--- | :--- |
| **0x01** (ascan) | **0x01** | `n_samples` | [15:0] Количество отсчётов |
| | **0x02** | `accum` | [7:0] Накопление |
| | **0x03** | `accum_type` | [1:0] Тип накопления |
| | **0x04** | `drop_ticks` | [15:0] Задержка регистрации (skip ticks) |
| | **0x05** | `pep_idx` | [1:0] Индекс выбранного ПЭП (коммутатор 1:4 приёмника) |
| **0x02** (pulse) | **0x01** | `charge` | [15:0] Время зарядки катушки |
| | **0x02** | `transfer` | [15:0] Время переноса энергии |
| | **0x03** | `strike` | [7:0] Время разряда на ПЭП |
| | **0x04** | `gen_mask` | [3:0] Маска подключения физических генераторов |
| **0x03** (tune)  | **0x01** | `start_amp` | [10:0] Начальное усиление |
| | **0x02** | `amp_one` | [31:0] Приращение до точки перегиба |

### План действий по реализации модуля `param`

1. **Шаг 1: Определение структуры внутренних теневых регистров (Shadow Registers)**
   * Создать многомерные массивы регистров `reg` в домене `sys_clk` с адресацией `[0:3][0:3]` (4 физических канала на 4 виртуальных цикла) для каждого параметра.
   * **Добавление нового параметра:** объявить массив теневых регистров `reg [15:0] r_sh_tune_drop_ticks[0:3][0:3]`.
   * Описать в блоке `always @(posedge sys_clk)` инициализацию всех 16 ячеек каждого параметра дефолтными безопасными значениями по системному сбросу `sys_rst_n`.

2. **Шаг 2: Реализация логики декодирования записи (Write Decoder)**
   * В блоке `always @(posedge sys_clk)` при наличии флага `i_cmd_vld == 1'b1`:
     * Извлечь координаты: `wire [1:0] w_cmd_phy = i_cmd_addr[15:14]` и `wire [1:0] w_cmd_vch = i_cmd_addr[13:12]`.
     * На основе ID модуля `i_cmd_addr[31:24]` и ID параметра `i_cmd_addr[23:16]` записать `i_cmd_data` в соответствующую ячейку `[w_cmd_phy][w_cmd_vch]` теневой памяти (включая запись в `r_sh_tune_drop_ticks` при ID параметра `0x07`).

3. **Шаг 3: Реализация указателя Ping-Pong буфера и двух активных банков**
   * Объявить 1-битный указатель `r_wr_ptr`. По сбросу `sys_rst_n` установить его в `0`.
   * Объявить две идентичные многомерные структуры активных параметров `bank_0` и `bank_1` (каждая размером $4 \times 4$ для каждого параметра, включая `drop_ticks` для `tune`).
   * По переднему фронту `i_sys_sync` (конец глобального цикла измерения):
     * Скопировать все параметры из теневых регистров в целевой банк `bank[r_wr_ptr]`.
     * Инвертировать указатель: `r_wr_ptr <= ~r_wr_ptr`.

4. **Шаг 4: Реализация коммутации реального времени для физических трактов**
   * Реализовать мультиплексоры в домене `sys_clk`, выбирающие из активного рабочего банка `bank[~r_wr_ptr]` параметры по индексу `i_sys_vch_sel` отдельно для каждого физического канала $P$ ($0..3$):
     * Считать параметры `ascan`, `pulse` и `tune` (включая `tune_drop_ticks`) для каждого $P$ из ячеек `[P][i_sys_vch_sel]`.
   * **Выходы домена `sys_clk` (Физические выходы быстрой коммутации):**
     * По фронту `i_sys_sync` зафиксировать параметры коммутации аналоговых каскадов ПЛИС (`o_ascan_chP_pep_idx` и `o_pulse_chP_gen_mask`) непосредственно в выходные регистры.
   * **Выходы домена `adc_clk` (CDC, Физические выходы `o_ascan_ch*`):**
     * В блоке `always @(posedge adc_clk)` при наличии синфазного импульса `i_adc_sync == 1'b1` защелкнуть выбранные параметры `ascan` для всех 4-х каналов в выходные регистры.
   * **Выходы домена `dac_clk` (CDC, Физические выходы `o_tune_ch*`):**
     * В блоке `always @(posedge dac_clk)` при наличии синфазного импульса `i_dac_sync == 1'b1` защелкнуть выбранные параметры регулировки усиления `tune` (включая `o_tune_chP_drop_ticks`) для всех 4-х каналов в выходные регистры.
   * **Выходы домена `hi_clk` (CDC, Физические выходы `o_pulse_ch*`):**
     * В блоке `always @(posedge hi_clk)` при наличии синфазного импульса `i_hi_sync == 1'b1` защелкнуть выбранные параметры генерации импульса `pulse` для всех 4-х каналов в выходные регистры.

5. **Шаг 5: Реализация двухкоординатного интерфейса чтения метаданных**
   * В блоке `always @(posedge sys_clk)` реализовать чтение из замороженного банка результатов `bank[r_wr_ptr]` на основе входных сигналов `i_packet_phy_ch` и `i_packet_vch`.
