### Анализ проблемы гонки параметров (Race Condition) в многоканальной системе

При использовании одинарного буфера активных регистров возникает критическая уязвимость в непрерывном цикле измерений:

1. **Запуск цикла N (`i_sys_sync` №1):**
   * Параметры из теневых регистров копируются в активные регистры.
   * Начинается физическое измерение №1. Модуль `ascan` записывает оцифрованные данные в первый буфер двойного буфера («бабочки») — **Буфер 0**.
2. **Окончание измерения №1:**
   * Запись в **Буфер 0** завершена. Выставляется флаг готовности `o_data_ready = 1`.
   * Однако хост-система или передатчик заняты (сигнал готовности шины чтения `i_out_rdy` равен `0`). Данные ждут отправки.
3. **Запуск цикла N+1 (`i_sys_sync` №2):**
   * Приходит следующий импульс запуска. Начинается измерение №2. Данные АЦП пишутся во второй буфер — **Буфер 1**.
   * **Проблема:** Если активные регистры в `param` одиночные, то по этому синку они мгновенно перезаписываются новыми значениями (для измерения №2).
4. **Считывание данных измерения №1:**
   * Наконец, передатчик готов (`i_out_rdy = 1`) и начинает вычитывать **Буфер 0** (измерение №1).
   * Вместе с данными он считывает метаданные (выходные порты `o_sys_ascan_*`, `o_sys_pulse_*` и `o_sys_tune_*`).
   * **Итог:** Передатчик прикрепит к физическим данным измерения №1 новые параметры от измерения №2. Произойдет искажение метаданных (разъюстировка параметров физического кадра, включая настройки ВРЧ).

---

### Решение: Двойной буфер активных регистров (Ping-Pong параметров)

Для полной изоляции параметров физического измерения от параметров считывания мы добавляем в модуль `param` **двухбанковую структуру активных регистров** (Bank 0 и Bank 1) и внутренний 1-битный указатель `r_wr_ptr`:

1. **Внутренний указатель записи `r_wr_ptr`:**
   * Инициализируется в `0`.
   * Инвертируется (переключается `0 -> 1 -> 0`) строго по каждому импульсу `i_sys_sync`.
2. **Запись параметров текущего измерения (Захват):**
   * По сигналу `i_sys_sync` параметры из теневых регистров копируются в банк, на который указывает `r_wr_ptr` (например, в `reg_bank[r_wr_ptr]`).
   * Из этого же банка (`r_wr_ptr`) берутся параметры для немедленной передачи в домены физического захвата (`adc_clk` и `hi_clk`) через CDC, а также для модуля регулировки усиления `tune` (`sys_clk`).
3. **Чтение параметров предыдущего измерения (Выдача результатов в пакет):**
   * Все выходные порты для формирования пакета результатов (`o_sys_ascan_*`, `o_sys_pulse_*` и `o_sys_tune_*`) подключаются к противоположному банку: `reg_bank[~r_wr_ptr]`.
   * Поскольку цикл чтения всегда отстает от цикла записи ровно на 1 шаг (пока мы записываем новые данные в Буфер 1, мы передаем ранее накопленные данные из Буфера 0), параметры для передачи гарантированно будут считаны из «замороженного» банка, который не изменится до следующего синка.

Это полностью разделяет контур реального времени (физические модули) и контур вывода данных (интерфейс чтения), вне зависимости от того, в каком тактовом домене работает целевой модуль.

### Обзор проектных решений для модуля `param`

1. **Архитектура теневых регистров (Shadow/Holding Registers):**
   Чтобы избежать ситуации, когда параметры меняются прямо посреди цикла измерения (что может привести к искажению данных A-скана или некорректной работе зондирующего импульса), модуль использует двухступенчатую структуру хранения. Запись по шине команд (`i_cmd_addr`, `i_cmd_data`, `i_cmd_vld`) в тактовом домене `sys_clk` всегда происходит только во временные теневые (Holding) регистры. Массив теневых регистров содержит **16 независимых наборов параметров** (по одному для каждого логического канала).

2. **Двойная буферизация активных регистров (Ping-Pong) для защиты от гонки данных:**
   В непрерывном режиме работы дефектоскопа новое измерение запускается по `i_sys_sync`, а данные предыдущего измерения еще могут находиться в процессе передачи в хост. Чтобы исключить перезапись метаданных кадра, модуль `param` реализует двухбанковую структуру активных регистров:
   * **Указатель `r_wr_ptr`:** 1-битный регистр в домене `sys_clk`, который переключается по каждому импульсу `i_sys_sync`. Он определяет, в какой банк будут сохранены параметры для нового измерения.
   * **Банк 0 (`bank_0`) и Банк 1 (`bank_1`):** Два идентичных набора активных регистров, каждый хранит настройки для 16 логических каналов.
   * **Логика работы:**
     * По сигналу `i_sys_sync` параметры из теневых регистров переносятся в банк, указанный `r_wr_ptr`.
     * На основе входных индексов коммутации (`i_sys_ch_sel_ch0` ... `i_sys_ch_sel_ch3`) из текущего рабочего банка извлекаются параметры для 4 параллельных физических трактов.
     * Параметры физических каналов (`o_ascan_ch*` в домен `adc_clk`, `o_pulse_ch*` в домен `hi_clk`, `o_tune_ch*` в домен `sys_clk`) уходят в работу.
     * Доступ к параметрам для формирования пакета результатов (`o_sys_ascan_*`, `o_sys_pulse_*`, `o_sys_tune_*`) осуществляется по запросу через индексный вход `i_packet_ch_idx`. Данные считываются строго из противоположного («замороженного») банка `~r_wr_ptr`.

3. **Безопасная передача параметров через тактовые домены (CDC):**
   Параметры передаются в несинхронные домены (`adc_clk` и `hi_clk`) по классической схеме **синхронизации квазистатических шин по импульсу запуска**:
   * В домене-источнике (`sys_clk`) мультиплексированные данные 4-х каналов фиксируются в выбранном банке по сигналу `i_sys_sync`.
   * В целевых доменах данные защелкиваются в локальные выходные триггеры по сигналам `i_adc_sync` / `i_hi_sync` (которые гарантированно приходят позже `i_sys_sync` из-за задержки CDC-цепочки в `rst_sync`). Поскольку к моменту прихода синков данные на шине из `sys_clk` уже стабильны, метастабильность исключена.

### module param

Модуль `param` принимает конфигурационные команды по входной шине в домене `sys_clk`, декодирует их, временно хранит в теневых регистрах, а по сигналу синхронизации копирует их в активную память. Модуль осуществляет коммутацию 16 логических каналов на 4 физических тракта и поддерживает пинг-понг буферизацию с индексным интерфейсом чтения метаданных для сборщика выходных пакетов.

### План действий по реализации модуля `param`

1. **Шаг 1: Определение структуры внутренних теневых регистров (Shadow Registers)**
   * Объявить массивы регистров `reg` размером `[0:15]` для каждого параметра в домене `sys_clk` (всего 16 логических каналов).
   * Реализовать инициализацию всех 16 ячеек каждого параметра дефолтными безопасными значениями по системному сбросу `sys_rst_n`.

2. **Шаг 2: Реализация логики декодирования записи (Write Decoder)**
   * В блоке `always @(posedge sys_clk)` при `i_cmd_vld == 1'b1`:

3. **Шаг 3: Реализация указателя Ping-Pong буфера и двух банков регистров**
   * Объявить 1-битный регистр указателя `r_wr_ptr`. По сбросу `sys_rst_n` инициализировать его в `0`.
   * Создать две структуры хранения активных параметров (Bank 0 и Bank 1), каждая из которых представляет собой массив из 16 элементов для всех параметров.
   * По переднему фронту `i_sys_sync`:
     * Скопировать все 16 наборов параметров из теневых регистров в целевой банк `bank[r_wr_ptr]`.
     * Инвертировать `r_wr_ptr <= ~r_wr_ptr`.
     * **Важно:** Физический тракт в текущем цикле будет работать с параметрами из свежезаписанного банка (который теперь доступен по индексу `~r_wr_ptr`), а пакетный вывод будет читать неизменный предыдущий банк по индексу `r_wr_ptr`.

4. **Шаг 4: Реализация коммутационной матрицы и CDC для физических трактов**
   * **Выборка параметров (домен `sys_clk`):**

   * **Домен `adc_clk` (Физические выходы `o_ascan_ch*`):**
